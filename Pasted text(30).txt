Senior Developer Review of the Remediation Plan
What the plan gets right
- Focuses on demo-impactful items first. The P0 pick of "broken back button" and "no logout endpoint" is exactly right — these are the first things a recruiter clicking through will notice.
- Correctly defers the large rewrites. No httpOnly cookie migration, no async scoring, no message queues. Appropriate restraint for a demo project.
- Dead code removal (B3) is smart. The RoleHierarchy bean is not only unused but misleading — any Spring Security consultant would flag it during a review. Removing it is 15 minutes well spent.
- C3 (clear query cache on logout) is a great catch. Most developers forget this, and stale cache data after logout is a real footgun.
What the plan gets wrong
1. Resume download is broken in the browser — not classified correctly
The current implementation uses <a href="...">Download Resume</a>. This is an anchor tag — the browser sends a GET request without the Authorization header because the JWT lives in localStorage, not in an httpOnly cookie. The result: clicking "Download Resume" in any browser tab opens a new tab that hits the backend without auth → returns 401/403 → shows blank page or error.
This is a P0 demo blocker — if a recruiter clicks a candidate's resume and gets a blank page, the demo is over. Your C4 fix addresses this (fetch via axios blob + programmatic download), but you marked it "Low impact." I'd raise this to High impact, Medium effort (2 hours for the download utility + wiring it into both ApplicationsPage and ProfilePage).
2. /admin route — the claimed fix is unnecessary
Re-reading the code: ADMIN clicking "Dashboard" in the sidebar goes to /admin. App.tsx routes /admin → DashboardPage. DashboardPage renders AdminDashboard for ADMIN users. This shows totalUsers, activeDrives, totalApplications via useStats(). That's correct admin content. No fix needed. Remove A5 from the plan.
3. Cross-role ADMIN access (B4) — the problem is already solved
You said: "ADMIN cannot access recruiter-only endpoints." But:
- GET /applications/job-post/{jobPostId}/ranked has @PreAuthorize("isAuthenticated()") + service-level role check using hasExactRole — admin has full access.
- GET /insights/* has @PreAuthorize("hasAnyRole('RECRUITER', 'PO', 'ADMIN')") — admin has full access.
- GET /analytics/* has @PreAuthorize("hasAnyRole('ADMIN', 'PO')") — admin has full access.
The only missing endpoint is GET /applications/job-post/{jobPostId} (non-ranked), but admin doesn't need it — the ranked endpoint provides richer data. This is already handled. Remove B4 — not a real gap.
4. Missing item: mutation loading states
When a PO clicks "Activate" on a drive, or a RECRUITER clicks "Shortlist" on a candidate — the button stays enabled, no spinner appears, and nothing visually confirms the action is in flight. The user can click twice, causing duplicate requests. These buttons exist in DriveDetailPage.tsx and ApplicationsPage.tsx. Adding disabled + Spinner to mutation buttons across the app is ~1 hour. Medium impact for demo polish.
5. Missing item: inconsistent API error surfacing
The backend returns ApiResponse.error(400, message) for BusinessException and ApiResponse.error(403, "Access denied") for AccessDeniedException. The frontend's Axios interceptor catches errors and shows toast.error(err.message). But err.message from axios is the HTTP status text ("Request failed with status code 400"), not the backend's ApiResponse.message. This means all backend validation errors surface as generic browser error messages instead of the actual business message.
Example: "You have already applied to this job" returns as "Request failed with status code 400" in the toast. This makes the app feel broken. Fix: parse the Axios error response body in the interceptor and extract the backend message. 30 minutes, Medium impact.
6. Effort estimate for C1 (OpenAPI) is light
Basic springdoc setup is 15 minutes. But useful Swagger documentation requires @Operation and @Schema annotations on endpoints and DTOs. For the key demo endpoints (auth, drives, job posts, applications, rankings), expect 3-4 hours for meaningful documentation, not 2.
